## 2. Что уже хорошо

### 2.1. Разделение public/private уже правильное
- Публичные страницы лежат в `/pages/`
- Приватные страницы лежат в `/app/`

Это хороший фундамент. Его не нужно ломать.

### 2.2. Есть здоровые data-first страницы
Самые устойчивые страницы уже работают по правильной модели:
- `pages/movies/`
- `pages/perfume/`
- `pages/recipes/`
- `app/finds/`

- `app/tele2/`
- `app/travel-wishlist/`

У них данные вынесены из HTML, и это сильно снижает риск случайных поломок.

### 2.3. Есть рабочие полуавтоматические сценарии
Особенно хороший пример — `pages/movies/`:

Например:

- если не получилось — локальный JSON

Это повышает живучесть страниц.

## 3. Основные проблемы

### 3.1. Сейчас существуют две разные системы

#### Система A — правильная
Данные лежат отдельно:
- JSON
- API
- отдельные служебные скрипты

#### Система B — хрупкая
Данные зашиты прямо в `index.html`:
- большие JS-массивы внутри страницы
- стили и логика тоже внутри страницы
- любое обновление требует ручной правки большого файла

Пока обе системы живут вместе, процесс создания новых страниц остаётся непредсказуемым.

### 3.2. Слишком много монолитных `index.html`
Во многих страницах всё хранится в одном файле:
- HTML
- CSS
- JS
- данные

Это было удобно на старте, но плохо подходит для роста.

### 3.3. Повторяются одинаковые UI-куски
Повторяются вручную:
- back button
- шрифты
- hero-блоки
- карточки
- фильтры
- общие сетки

Следствие:
- сложно поддерживать единый стиль
- одно улучшение надо повторять во многих местах
- легко получить «почти одинаково, но уже разъехалось»

### 3.4. Есть признаки аварийного режима поддержки
В `pages/perfume/` лежат резервные копии вроде:
- `index_bag_2026-02-14_1250.html`
- `index_bag_2026-02-14_1615.html`
- и т.д.

Это понятная защитная реакция, но как системный процесс — плохой знак. Значит, менять страницу страшно.

### 3.5. Нет одного общего контракта для новых страниц
Пока не зафиксированы единые правила:
- когда страница public, а когда private
- где обязателен JSON
- как называть папки
- где хранить картинки
- как подключать страницу в главный хаб
- как безопасно обновлять существующую страницу

## 5. Базовый стандарт на будущее

Новые страницы делать только по одной схеме.

### 5.1. Структура папки

#### Public page
```text
/data/workspace/pages/<slug>/
  index.html
  data.json
  img/
```

#### Private page
```text
/data/workspace/app/<slug>/
  index.html
  data.json
  img/
```

### 5.2. Что где хранится
- `index.html` — только интерфейс и логика отображения
- `data.json` — контент
- `img/` — локальные изображения страницы
- скрипты импорта/обновления — рядом, только если реально нужны

### 5.3. Правило данных
Если страница — каталог, подборка, список, коллекция, план, wishlist или база объектов:
- данные **не вшиваются в HTML**
- данные идут в `json`

### 5.4. Правило добавления
Чтобы добавить новую страницу, должно хватать 4 действий:
1. создать папку
2. положить `index.html`
3. положить `data.json`
4. добавить ссылку в хаб (`pages/index.html`) при необходимости

## 8. Короткий вывод

Сейчас система страниц **уже рабочая**, но ещё не полностью стандартизирована.

Главная задача не «переписать всё с нуля», а:
- сохранить то, что уже работает
- перестать создавать новые страницы в ручном/хаотичном режиме
- постепенно перевести хрупкие страницы на data-first формат

То есть курс правильный такой:
**не ломать → стандартизировать → потом спокойно упрощать**.
